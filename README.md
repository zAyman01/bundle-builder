# Security System Bundle Builder

A data-driven, two-column bundle builder: a 4-step accordion on the left, a live
review panel on the right, built to match the shared Figma/Anima export.

## Run it

```bash
npm install
npm run dev      # http://localhost:5173
```

```bash
npm run build     # production build to /dist
npm run preview    # serve the production build locally
```

Requires Node 18+.

## Stack

- React 19 + TypeScript, Vite
- Tailwind CSS v4 (via `@tailwindcss/vite`, no separate config file â€” theme
  lives in `src/index.css`)
- No external state library â€” a single `useBundleState` hook drives everything
- `localStorage` for "Save my system for later" persistence

## How it's structured

```
src/data/bundle.ts          - the JSON-like source of truth (products, plans, seed state)
src/types/index.ts          - Product / Variant / PlanOption / Step types
src/hooks/useBundleState.ts - all app state: quantities, active variant per
                              product, selected plan, open accordion step,
                              derived pricing, save/restore
src/components/
  StepAccordion.tsx         - one accordion step (header + product grid or plan cards)
  ProductCard.tsx           - a single product: badge, image placeholder,
                              variant chips, stepper, pricing
  QuantityStepper.tsx       - shared +/- control used on cards AND review lines
  ReviewPanel.tsx           - "Your security system" summary panel
  icons.tsx                 - small inline SVGs (chevrons, +/-, category icons)
```

Nothing renders per-product JSX â€” every card and review line is generated by
mapping over `steps` / `reviewLines`. Add a product by adding a JSON entry to
`bundle.ts`; no component changes required.

## Key interactions

- **Variant-aware quantities.** Each variant (`productId:variantId`) has its
  own count in `useBundleState`. Selecting a color only changes which variant
  is "active" for that card's stepper â€” it never touches other variants'
  counts. The review panel lists every variant with qty > 0 as its own line
  (e.g. switching a card to Blue after adding 2 Red still shows "Red (Ă—2)").
- **Synced steppers.** The card's stepper and the review panel's stepper for
  the same line both read/write the same state slice, so either one updates
  the other instantly.
- **"N selected"** counts distinct *products* with any variant quantity > 0
  (not total units), per the spec.
- **Save my system for later.** Deliberately *not* auto-persisted on every
  keystroke â€” the button snapshots the current quantities/variants/plan into
  `localStorage` only when clicked, which is what "configure â†’ save â†’ leave â†’
  return â†’ restored" actually asks for. Reloading without ever clicking Save
  starts over from the seed state, matching the letter of the spec.

## A pricing decision worth calling out

In the export, the review panel's shown dollar amount for a multi-quantity
line (e.g. Wyze Cam Pan v3 Ă—2 â†’ "$47.98") is a static mock value that isn't
actually `unit price Ă— quantity` from the same product's card (2 Ă— $34.98 =
$69.96, not $47.98). Since the spec explicitly requires the total to
*recalculate correctly* as quantities change, I treat each product's card
price as the canonical per-unit price and always compute
`line total = unit price Ă— quantity` (and grand total = sum of one-time line
totals + one month of the selected plan). This means the **initial** dollar
figures shown differ slightly from the exact static numbers in the mock, but
every number in the app is now internally consistent and behaves correctly
under any quantity/variant change â€” which I judged to matter more than
matching a mock's non-multiplied placeholder numbers exactly.
